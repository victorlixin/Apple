Mixed-mode Programming Notes

 

This section describes issues of importance to developers of mixed-mode applications. Actions prescribed in some of the calls are required for mixed-mode applications to compile and run successfully.

 

  1. Any Mixed mode program must exec after a fork() call, to continue to use mixed mode features.

The forked child process can make only SUA calls. In a process fork, child processes do not use Windows resource handles and therefore cannot make Win32 calls.

 

  1. Signal-related issues

Mixed-mode programs must handle any SUA signal handler invocations, even when executing .dll code. Specifically, any calls made to alertable Windows system APIs must return values accordingly.

 

  1. Do not mix MSVCRT and SUA C library

The mixed mode application must not link to both MSVCRT and SUA C libraries (libc), because linking can fail due to library name conflicts. Instead, create a wrapper library that exposes the functionality you want from MSVCRT, and link from an appropriate library within your mixed mode application to the SUA C library.

 

  1. Console handling

Mixed-mode programs must perform all console handling operations only in the SUA portion of the code.

 

  1. Use N nostdc option while using ODBC headers shipped with Microsoft Visual C++

When you are linking with ODBC headers for example, sqltypes.h use the N flag with the nostdc option to enable compilation. This is because headers shipped with Microsoft Visual C++ are not ANSI-based files.

The following is an example of how to use this option.

c89 N nostdc

 

  1. Y option is not recognized during compilation of an SUA DLL

When you are compiling code into an SUA .dll, do not use the -Y flag to specify the .dll entry point in the SUA .dll. During the SUA dll build process, the linker expects DllMain() to be the entry point and fails if DllMain() is not found.

 

  1. Usage of SetUnhandledExceptionFilter in a mixed mode program is not allowed

A mixed mode program should not use SetUnhandledExceptionFilter() Win32 API (either directly, or inside the .dll to which the program links). If it does, signal handling might fail.

 

  1. Mismatch in data type length between UNIX64 and Windows64 architecture can lead to data corruption in mixed-mode applications

This issue typically occurs wherever the long format is used in the UNIX code. (long in UNIX is 64-bit; in Windows it is 32-bit). Wherever there is a requirement for 32-bit data in Windows, convert the long to int, making the code portable between UNIX and Windows platforms.

 

  1. Signals might not work properly in programs with OCI calls

Signals do not work correctly in mixed-mode applications primarily because of differences between the implementations of liboci.a and oci.dll. The Windows OCI .dll depends on blocking and non-alertable function calls.

Use the following general guidelines for rewriting code to take advantage of mixed mode features:

         For single-threaded apps, OCI calls should be made in non-blocking mode. For blocking behavior, you can loop OCI calls, though this will have a performance impact.

         For multi-threaded applications, block signals on all threads that have OCI calls. In this case, the signals would get delivered in the non-OCI threads without issues. If there is no non-OCI thread, the developer can create one with code as shown in the following example:

while(1) pause();

 

  1. Differences between advisory and mandatory locks

UNIX semantics are advisory in nature while Windows locking is mandatory in nature. Calling both advisory and mandatory locks in the same mixed-mode program is not supported.

 

  1. Asynchronous cancellation does not work in pthreads / Win32 threads during Sleep or SleepEx (Win32 API)

In a mixed-mode application, when the control is inside Sleep or SleepEx, asynchronous cancellation of a thread (both pthread and Win32 thread) fails.

Note: Signaling also fails when a thread is inside Sleep or SleepEx. In other words, the signals do not make the thread resume from the sleep phase.

 

  1. Windows style paths required for OCIFile APIs

The OCIFile APIs of the OCI .dll can process only Windows-format paths. For example, a path in POSIX format, such as /bin/xyz is not accepted by OCIFile APIs.

Notes:

         You can use posixpath2nt to convert UNIX paths to Windows-formatted paths.

         Spfile names are also operating system-dependent. If you are using them from PL/SQL blocks in Oracle, modify the paths to the correct Windows location.

 

  1. Environment variable related Win32 APIs and SUA APIs should not be mixed

In a mixed mode program, the GetEnvironmentVariable and SetEnvironmentVariable should not be mixed with setenv and getenv. In other words, if you set an environment variable using Win32 API, retrieve it using the same Win32 API set.

 

  1. Win32 threads do not inherit signal masks from the parent thread

In a mixed mode application, when a new Win32 thread is created, signal masks are not inherited from the parent thread. Signal masks should be reinitialized as soon as the thread is created.

 

  1. Changes required for Enabling ODBC applications

a.    Include mixedmode.h for compiling all ODBC programs. Mixedmode.h includes standard declarations which are required for ODBC programs to compile.

 

b.    Copy the below odbc header files to \usr\include\odbc

Source location: <DirveLetter>\Program Files\Microsoft Visual Studio 8\VC\PlatformSDK\Include

odbcinst.h

odbcss.h

sql.h

sqlext.h

sqloledb.h

sqltypes.h

sqlucode.h

 

c.    Replace all instances of long int with int in sqltypes.h. Then replace all instances of long with int.

Note:

                                                          i.    Do the find replace operation in the order mentioned above. If you first search for long and replace it with int, then long int will be replaced by int int which is an invalid data type.

                                                         ii.    Do a case sensitive search for long int and int respectively.

                                                        iii.    Have a backup of sqltypes.h before you modify the file.

 

d.    Comment out the sql_api definition from %programfiles%\Microsoft Visual Studio 8\VC\PlatformSDK\Include\sqltypes.h

 

#ifdef WIN32

#define SQL_API  __stdcall

#else

#define SQL_API

#endif

 

  1. Provide a complete path when linking to a DLL (through a .lib file) by using gcc

When gcc is used to compile and link a mixed mode program, provide the complete path for the library files.

Example:

gcc R test.c ./testfunc.lib o test

where testfunc.lib represents the DLL to which the code is linked. Note the ./ before the library name.

 

  1. OCI program in shared mode is not supported

OCI programs using OCI_SHARED mode are not supported, and will return an error in SUA.

 

  1. Win32 programs built outside of SUA that perform mixed-mode calls to SUA libraries must link to mixcrt0.o

All win32 applications that are built without using c89 or SUA must link to mixcrt0.o if these applications make mixed-mode calls to SUA libraries.

 

  1. Mixed mode applications cannot impersonate users

Setuid, seteuid, setuser and other calls that change the user identity of the process fail when called from a mixed-mode application. Mixed-mode applications cannot be marked as setuid applications.

64-bit Compilation Notes

Disabling File System Redirection (FSR)

In 64-bit computers, any access to %windir%\System32 is automatically redirected to %windir%\syswow64, if there are 32-bit processes that run on WOW64. If you need to use the system32 directory in a 32-bit thread, use the pdxWow64EnableFSRedirection API. The following documents the pdxWow64EnableFSRedirection API.

 

pdxWow64EnableFSRedirection () enable/disable file system redirection from system32 to SysWOW64

 

 

pdxWow64EnableFSRedirection () enable/disable file system redirection from system32 to SysWOW64

 

 

SYNOPSIS

 

   #include <interix/interix.h>

 

   int pdxWow64EnableFSRedirection(int redir)

 

DESCRIPTION

 

The pdxWow64EnableFSRedirection(2) function

 

1.    enables the FSR (file system redirection) if redir is ENABLE_REDIRECTION

2.    disables the FSR if redir is DISABLE_REDIRECTION

   for a 32-bit process on a 64-bit computer.

 

When FSR is enabled, all calls to files in %windir%\system32 are redirected to the %windir%\SysWOW64 process for a WOW64 thread.

 

This api exempts the following four paths from redirection:

 

1.    %windir%\system32\drivers\etc

2.    %windir%\system32\spool

3.    %windir%\system32\catroot

4.    %windir%\system32\catroot2

  

   FSR is enabled/disabled on a per-thread basis.

 

FSR is enabled by default for a thread, Here, the main thread of a child process during a fork may be an exception which inherits this property from the parent thread.

 

RETURN VALUES

 

   EINVAL : when redir is neither ENABLE_REDIRECTION nor DISABLE_REDIRECTION

   EZERO  : success

   EACCES : when the function is called from a thread that is not WOW64-compliant.

 

 

 

 

Notes

1.    File System Redirection fails when %windir%\System32 is accessed as %windir%\system32

On 64-bit computers, any access to %windir%\System32 is automatically redirected to %windir%\syswow64, in case of 32-bit processes running on WOW64. Redirection is case-sensitive, and fails if the System32 is specified as system32.

2.    Cross-compilation support

To perform a 64-bit compilation on a 32-bit computer, or vice-versa, set up the correct environment variables for the compiler, linker and lib. The required environment variables are:

a.    C89_COMPILER - This variable must point to the correct compiler (cl.exe) version. For example, on an x86 computer, if you want to compile for AMD64 , point this variable to a compiler that runs on x86 but generates AMd64 code.

b.    C89_LINKER - Same as above, but this is for the link.exe.

c.    AR_LINKER - Same as above, but this is for the link.exe.

d.    AR_LIBRARIAN - Same as above, but this is for the lib.exe.

As an alternative, set the PATH variable to point correctly to the above variable locations, so that the correct cl.exe, lib.exe and link.exe are incorporated.

 

Notes:

         /usr/lib contains 32-bit libraries on a 32-bit computer and 64-bit libraries on an AMD64 computer.

         ar script in the 64-bit environment is a script wrapper over lib.exe, and it supports only a limited set of options.

         64-bit compilation supports the LP64 data model.

         Read or write fails for a request on greater than 4 GB of data. To work around this issue, split the request into several smaller requests.

12. Use of fork in a multi-threaded process can cause unexpected results

Assume a multi-threaded application containing threads A and B such that A is calling fork and at the same time B has acquired NTDLL heap lock. The child process is created. The NTDLL heap lock is marked as acquired by thread B. There is no thread B in the child. The main thread in the child while coming up tries to take the NTDLL heap lock, and because of the librarys locked state, the process would stop responding.

This issue is valid for both mixed-mode and Subsystem for UNIX-based Applications programs.

 

13. Pthread_Create() does not support stack-address attribute

POSIX.1 standard specifies that applications calling pthread_create() can specify the stack that the new thread should use by setting the stack-address attribute of pthread_attr_t structure (using pthread_attr_setstack) that is passed to Pthread_Create call as an argument. The pthread_Create() call available with this version of SDK does not support this attribute. Even if the stack-address attribute of pthread_attr_t structure is set by an application, pthread_create ignores the attribute, and the thread created receives a system-generated stack.

16. Enabling or Disabling setuid behavior for SUA programs

 

According to the POSIX standard, a file has permissions that include bits to set a UID (setuid) and set a GID (setgid) when the file is executed. If either or both bits are set on a file, and a process executes that file, the process gains the UID or GID of the file. When used carefully, this mechanism allows a non-privileged user to execute programs that run with the higher privileges of the file's owner or group. However, this can present security risks by allowing non-privileged users to perform actions that should only be performed by an administrator. For this reason, Microsoft Utilities and SDK for Subsystem for UNIX-based Applications Setup does not enable support for this mechanism by default.

 

You should enable support for setuid behavior only if you run programs that require support for this behavior. If you do not enable support for setuid behavior when installing Microsoft Utilities and SDK for Subsystem for UNIX-based Applications, you can enable it later.

 

Important: Incorrectly editing the registry may severely damage your system. Before making changes to the registry, you should back up any valued data on your computer. Microsoft cannot guarantee that problems resulting from using Registry Editor incorrectly can be solved. Use Registry Editor at your own risk.

To enable setuid

1.    Click Start and then click Run, (or, if you are running Windows Server 2012 or Windows 8, click in the Start Search text box).

2.    Type regedit in the Open text box on the Run dialog box, or in the Start Search text box.

3.    In the hierarchy pane, navigate to HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\SUA.

4.    Double-click EnableSetuidBinaries.

5.    In the Edit DWORD (32-bit) Value dialog box, enter 1 in the Value data text box.

6.    Click OK. Close Registry Editor and save your changes if prompted.

To disable Setuid

1.    Click Start and then click Run, (or, if you are running Windows Server 2012 or Windows 8, click in the Start Search text box).

2.    Type regedit in the Open text box on the Run dialog box, or in the Start Search text box.

3.    In the hierarchy pane, navigate to HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\SUA.

4.    Double-click EnableSetuidBinaries.

5.    In the Edit DWORD (32-bit) Value dialog box, enter 0 in the Value data text box.

6.    Click OK. Close Registry Editor and save your changes if prompted.

 

Notes:

         Administrator privileges are required to enable or disable setuid.

         You must restart the computer for your changes to take effect.

         Setuid bit usage is not advisable in the current release, since this has a security impact.

Moving Setuid files

A setuid binary fails if either of the following actions is performed:

         A file marked Setuid is moved from one location to another

         A system restoration is performed after deleting the setuid file to a restore point

To solve this problem, reset the setuid bit using the chmod command.

 

17. Case sensitivity and system security

 

During installation of Microsoft Utilities and SDK for Subsystem for UNIX-based Applications, you can choose to change the default behavior of object names such as file names to be case-sensitive. Case sensitivity affects system security as well as how Subsystem for UNIX-based Applications functions.

In Windows, the names of most objects (such as files and directories) are case-preserving, but case insensitive. That means you can use uppercase and lowercase characters when naming such objects, but Windows does not distinguish between names based on case alone. For example, you cannot have two files in the same directory named sample.txt and Sample.txt because Windows regards the names to be identical for the purposes of identifying files. The UNIX operating system, on the other hand, is fully case-sensitive, and UNIX-based computers distinguish between object names when the only difference between those names is the case of characters used in the object names. On most UNIX-based computers, therefore, sample.txt and Sample.txt could appear in the same directory, and the UNIX computer would distinguish between them when performing operations on the files. For example, the command rm S*.txt would delete Sample.txt but not sample.txt.

 

To implement typical UNIX behavior, SUA is normally case-sensitive when working with file names. This can present security issues, particularly for Windows users who are accustomed to the case-insensitive conventions of Windows. For example, a Trojan horse version of edit.exe named EDIT.EXE could be stored in the same directory as edit.exe. If a user were to type edit at a Windows command prompt, the Trojan horse version (EDIT.EXE) could be executed instead of the standard version. If case-sensitivity is enabled, Windows users should be made aware of this possibility.

 

In Windows Server 2012 And Windows 8, the default behavior of subsystems other than the Win32 subsystem is to be case-preserving but case-insensitive; in previous versions of Windows, such subsystems were fully case sensitive by default. In order to support standard UNIX behavior, you have to do the following to enable case sensitivity.

Warning: If you use Registry Editor incorrectly, you might cause serious problems that may require you to reinstall your operating system. Microsoft cannot guarantee that you can solve problems that result from using Registry Editor incorrectly.

To enable case sensitivity

1.    Click Start and then click Run, (or, if you are running Windows Server 2012 or Windows 8, click in the Start Search text box).

2.    Type regedit in the Open text box on the Run dialog box, or in the Start Search text box.

3.    In the hierarchy pane, navigate to HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Session Manager\kernel.

4.    Double-click obcaseinsensitive.

5.    In the Edit DWORD (32-bit) Value dialog box, enter 0 in the Value data text box.

6.    Click OK. Close Registry Editor and save your changes if prompted.

To disable case sensitivity

1.    Click Start and then click Run, (or, if you are running Windows Server 2012 or Windows 8, click in the Start Search text box).

2.    Type regedit in the Open text box on the Run dialog box, or in the Start Search text box.

3.    In the hierarchy pane, navigate to HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Session Manager\kernel.

4.    Double-click obcaseinsensitive.

5.    In the Edit DWORD (32-bit) Value dialog box, enter 1 in the Value data text box.

6.    Click OK. Close Registry Editor and save your changes if prompted.

 

Notes:

         Administrator privileges are required to enable or disable case sensitivity.

         You must restart the computer for your changes to take effect.

 

18. Applications that link to psxlibc.dll programs cannot declare libc global variables as extern.

For example the libc global variable tzname (for timezone name information) cannot be used as extern as follows -

 

#include<time.h>

extern char *tzname[];

 

This cannot be done as linking to psxlibc.dll causes the declaration of libc globals as getter functions (due to the security and functional issues with exposing global variables from a DLL). For example for the libc global variable tzname the following declaration takes effect when linking to psxlibc.dll -

 

#if !defined (_IMPORT_LIBCDLL_GLOBALS)

extern char* tzname[2];

#else

#include <globalexport.h>

#define tzname ((char**)libc_global(G_TZNAME))

#endif

 

This defines the symbol tzname a as a char** function which would cause a mismatch in the above application code which declares tzname as an extern char*[] variable.

 

Due to such issues declaration of libc global variables as extern in a application that linksto psxlibc.dll is not supported.

The man pages for the SVR-5 utilities include references to file paths like /bin , /usr etc. Except for /dev and /proc, all paths mentioned in the SVR-5 utilities man pages should have /svr-5 prepended. Daemons do not run in session zero if they are started by a user

 

Daemons, when started at system startup, run in session zero. If daemons are started manually, they start in the current users active session rather than session zero. The following results:

         Daemons that run in a session other than session zero stop once the session in which they are executing is closed. A session ends when the user logs off.

         When rsh,rlogin,telnet,cron daemons are started in a session other than session zero(0), Win32 or mixed mode applications launched by non administrator users fail if started by using rsh/rlogin/telnet/cron.

 

Daemons can be restarted in session zero by using any of the following methods.

1.    Restart daemons by sending SIGHUP signal twice to init daemon (to switch init to down state and then back to up state). Note: This method restarts all running daemons.

2.    Restart the daemons in a rsh/telnet/rlogin session or by using cron. This will apply only if rsh/telnet/rlogin themselves are running in session 0.

3.    Restart the computer.

Appendix C: .so to .dll conversion

 

A conversion of .so to .dll could be required if it is necessary to migrate from the GNU SDK to c89-based compilation.

Note: It is not possible to link to a .so file and a SUA .dll simultaneously, because.so files link to libc.so, while SUA .dlls link to libc.dll. An application cannot link to two C runtimes without symbol conflicts. Linking to two C runtimes in the same application is not supported by SUA.

Following are the steps necessary to convert a .so file into a .dll file.

1.    Extract all the symbols (functions and variables) from the object files of the .so file. This can be done using the link.exe program. Refer to the link.exe help in MSDN for more details.

2.    Create a .def file out of this (Refer to MSDN for a discussion on .def files.)

3.    Because global variables from .so are not directly exposed, add code similar to the following to header files in your main program.

In this example, xyz is the variable exposed from the .so file, and the following code would be added to the program:

 

#ifdef BUILD_USING_DLL

#define xyz (*glob_ xyz())

#elseif

extern int xyz;

#endif

 

4.    Expose the following kind of function from the DLL to retrieve the variable.

 

glob_ xyz()

{

return & xyz;

}

 

5.    Compile related OBJ files of the .so file with your.def file by using the G B dll option to generate a .dll. See the previous section for a discussion on B and G options.
